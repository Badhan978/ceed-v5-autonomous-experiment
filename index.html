V5 完全自走実験 — PDCA 記録 / 実験チャーター

最終更新: 2026-06-30 設計骨格 完成（6 セクター mandate 確定） 未発火 Team 共有用

完全自走型 v5 の feasibility probe。人間は普遍の Goal（売上 / Imp 目標）だけを渡し、以降は機械的なデータ受け渡しを除きゼロ介入で、AI セッションが 6 セクターの役割を横断して PDCA ループを自走で回す。本ドキュメントはその設計・判断・サイクル記録。

0. これは何か

完全自走型 v5 の feasibility probe（信号が出るか見る一発）。人間は普遍の Goal（売上 / Imp 目標）だけを渡し、以降は機械的なデータ受け渡しを除きゼロ介入で、AI セッションが下記 6 セクターの役割を横断して PDCA ループを自走で回す。本ファイルはその設計・判断・サイクル記録を残し、Team へ共有するためのもの。

1. 仮説と背景（Why）

  • 位置づけ = v5 を根本から疑う。 現状コア AI が自分で PDCA を回した成果を感じられておらず、helper tool に留まっている。
  • 律速の診断: ①人間の中途半端な介入 ②動画パイプライン（Playbook）の制約 ③人間側の知識・ナレッジの天井。
  • 賭け: 計算力・保有情報・莫大な外部データへのアクセス権を持つ Claude に、「設計 → 毎サイクルの方向調整 → Goal 前進」を丸ごと任せたらどこまで行くか。これは LLM が本来得意とする領域。
  • probe の性格: 新規コールド垢・制約フリー・最小スキャフォールド。「対照付き証明」ではなく「信号が出るか見る一発」。後で human-in-loop と比較したくなったら別途 control が要る。
  • 設計指針: #2/#3（コンテンツ生成）を細かく仕様化しない。作り込むと逃れたかった Playbook 制約を再建し "制約フリー自走" の thesis に反する。ルール（#5）とデータ契約（#4）はハードに固め、戦略・生成は mandate（役割・入力・出力・guardrail）だけ渡して AI に委ねる。

2. 凍結した Goal spec（D1 locked: 2026-06-30）

項目内容
商材メーカー転職ならタイズ（A8 / category「仕事情報」/ payout_type fixed）
CV 発火点新規会員登録完了時点（ラグなし。否認は不正・不備・虚偽・いたずら・重複・即時解約・ログイン無・リスティング違反のみ）
単価¥10,000 / CV → 1 CV = 月間目標
目標売上 ¥10,000 / 月（directional）、最終射程 ¥50,000
対象新規コールド IG 垢・制約フリー
観測窓1 ヶ月
AI が舵を切る上流信号reach → クリック → 会員登録

選定根拠: 成果条件が純粋な上流発火（会員登録完了時点）で、GEO / 年齢 / 連絡確認 / 経験の縛りが一切ない唯一のオファー。¥10,000 = 1 CV で目標到達するため、コールド垢の volume 問題が「1 ヶ月で登録 1 件取れるか」に最小化される（最も forgiving）。弱点は TAM がメーカー / ものづくり志望に寄る点だが、コールド新規垢は 1 ニッチに深く張る方がアルゴリズム的に有利で、競合の薄い実在の鉱脈。

却下したオファーと理由:

  • ANNA（¥7,500）— 成果条件が「登録後 30 日以内に連絡が取れた時点」= 最大 30 日ラグ + 連絡不通否認 + GEO/年齢縛り。
  • レバテック（¥10,000）— 現役 IT / 未経験除外で TAM 狭、cookie 計上が脆い。
  • ワンキャリア（¥3,000）— クリーンだが正社員経験必須 + 約 4 CV 必要。最広 TAM の対抗馬。
学び（一般則）: 人材紹介は payout ラベルが「会員登録」でも成果条件に連絡 / 実施依存が隠れる。発火点は payout 文字列でなく成果条件で読む。

3. 測り方（指標構造）

  • 主指標 = 上流ファネル軌跡（reach → クリック → 会員登録）。理由: コールド 1 ヶ月では売上は volume 枯れ + 初速ノイズに支配され、自走 PDCA の質を測れない。週次の方向調整はこの上流信号で回す。
  • 副指標 = 売上（directional・月次チェック）。1 CV = ¥10,000 = 目標。
  • 原則: 売上を pass/fail にしない。未達を「自走失敗」と即断しない（コールド垢 1 ヶ月の confound を切り分けられないため）。
  • 3 段 tier: floor = ファネル成立 + クリック/CV 軌跡が上向き / target = ¥10,000 / stretch = 最終 ¥50,000 への射程が見える。

4. システム構成 — 6 セクターモデル

Session は毎発火でこの 6 セクターを役割として横断し、PDCA を回す。依存関係:

[#5 制約・ルール]  ← 絶対の枠（全セクターを拘束）
        │
   [#0 全体戦略]    ← 統括・絶対効力（#1–#3 の指針を決める）
    ├ #1 アカウント戦略
    ├ #2 集客コンテンツ生成
    └ #3 アフィ送客コンテンツ生成
        ↑
   [#4 データ分析]   ← #0–#3 へデータ供給

設計方針: #5・#4 はハード確定（人間所有）。#0–#3 は mandate レベル（役割・入力・出力・guardrail のみ、中身は AI に委ねる）。

#0 全体戦略（アカウント・アフィ）— 統括 / 絶対効力✅ mandate 確定

  • 権限: Session 内の最上位。#1–#3 への指針を決め、その判断は #1–#3 に対して絶対。ただし #5（制約・ルール）には従う。
  • 入力（毎発火で読む）:
    • Goal spec（§2）= 不変の北極星（¥10,000 / CV、上流信号 reach → クリック → 会員登録）
    • 口座×日サマリ = 数値トレンド + 視聴層属性（年齢 / 性別 / 国）・流入元・初速
    • 投稿単位の eval / bottleneck / verdict 集約 = 何が効いた / 効かない
    • 手動 CV ドロップ = アフィの刺さり具合（ground truth）
    • 前サイクルの実験台帳 = 自分の前回仮説・予測 vs 実測
  • 毎発火で答える問い（reasoning mandate）:
    1. 現状は Goal に対しどこ? 上流信号（reach / クリック / 登録）は伸びているか?
    2. 得られている視聴層は誰で、狙うべきユーザーニーズと合っているか?
    3. アフィ（メーカー転職）はその層に刺さっているか? ズレなら「層」か「訴求」どちらを動かすか?
    4. 前回の仮説は当たったか? PDCA サイクル自体を再定義すべきか?
  • 出力: 「一言の指針」（#1–#3 が従う）/ 根拠 + 検証中の仮説を実験台帳に記録（予測を明記）/ kill・success・scale 判定（該当すれば人間に報告）
  • guardrail: #5 に従う / 数値だけ追って視聴層・ニーズを無視した最適化をしない（ノイズ追従の戒め）/ 1 サイクルで戦略を全部ひっくり返さない（thrash 防止）= 変更は仮説ベースで、予測 → 実測で検証できる粒度に

#1 アカウント戦略✅ mandate 確定

  • 役割 / 権限: #0 の指針を「アカウント運用レベル」に翻訳する中間層。どこを調整するかを決め、#2 / #3 が実行できる施策に落とす。
  • 毎発火で答える問い:
    1. #0 の指針を実現するに、運用のどこを動かす?（集客 #2 とアフィ送客 #3 の比率 / 狙う層の絞り・広げ / 投稿時間・フォーマット）
    2. ファネルのどこが詰まっている?（TOFU 到達 / MOFU 関心 / BOFU 行動）→ ボトルネックに資源を寄せる
    3. 今は集客フェーズかアフィ刈り取りフェーズか?（新規垢の成長段階で #2 / #3 の重心を決める）
  • 出力: #2 / #3 への施策指示 =「何を・どの比率で・どの層 / 切り口を狙って」作るか / 調整の意図（仮説）を実験台帳へ
  • guardrail: #0 の指針に反しない / #5 に従う / アカウントの一貫性（視聴層・世界観）を壊す急旋回をしない / #2 / #3 の表現の中身（How）までは縛らない（What とどこを狙うかだけ渡し、作り方は委ねる = 制約フリー保持）

#2 集客コンテンツ生成（Reel / カルーセル）✅ mandate 確定

  • 役割: #1 の施策指示を受け、集客用コンテンツ（シナリオ → 動画）を生成・投稿。目的 = Imp 最大化 + 狙った層をページ（プロフ / リンク）へ送客。自己 PDCA で常時最適化。
  • 毎発火で行うこと:
    1. 過去投稿の eval / bottleneck を読み、何が効いた / 外したかから次の仮説を立てる
    2. 狙う層・ニーズに刺さる Hook → Body → CTA を設計（表現・言い回し・タイミング・誰に向けてを熟考）。CTA は集客用（リーチ増幅型中心）
    3. シナリオ → 動画生成 → ホスティング → 投稿キューへ
    4. 投稿後、自分の投稿に eval / bottleneck / verdict を記入（#4 へ書き戻し）+ 仮説と予測を実験台帳へ
  • 出力: 投稿 / 各投稿の eval・bottleneck・verdict（DB 書き戻し）/ 集客の仮説・予測（実験台帳）
  • guardrail: #1 の狙いと #5 に従う / 集客とアフィ送客を混同しない（#2 は Imp と送客が目的、直アフィ訴求は #3）/ Playbook 要点（ひらがな読み上げ・Hook/Body 教訓）は "参照" として尊重するが、Playbook が作れるものに自分を縛らない（必要なものを必要に作る = 制約フリー）

#3 アフィ送客コンテンツ生成（Stories / カルーセル / Reel）✅ mandate 確定

  • 役割: #1 の施策指示を受け、アフィ訴求特化のコンテンツを生成・投稿。目的 = 既存フォロワー + #2 で来た新規ユーザーをファネルへ接続（プロフ → bio / Story リンク → ASP 会員登録）。自己 PDCA で最適化。
  • 毎発火で行うこと:
    1. 過去のアフィ送客投稿の eval +「リンククリック → CV」の歩留まりを読み、何が刺さった / 落ちたかから仮説を立てる
    2. メーカー転職の訴求を、#2 で集めた層の痛み / ニーズに接続する設計（CTA プロフ誘導型中心、MOFU → BOFU）
    3. Stories / カルーセル / Reel を生成 → 投稿
    4. eval / bottleneck / verdict 書き戻し + 仮説・予測を実験台帳へ。CV と照合
  • 出力: アフィ送客投稿 / eval・bottleneck・verdict（DB）/「送客 → CV」の仮説・予測（実験台帳）
  • guardrail: #1 / #5 に従う / 温まっていない層に押し売りしてフォロワーを焼かない（ファネル順序を守る）/ アフィ訴求の頻度・濃度がアカウントの一貫性・reach を壊さないよう #0 / #1 のフェーズ判断に従う

#4 データ分析 — データ契約確定✅ 確定

  • 役割: #0–#3 が使うデータを収集し分析可能に整形。読むだけでなく、各サイクルの分析結果（eval / bottleneck / verdict）を DB に書き戻すのも #4 の仕事 = 自走 PDCA のメモリ。
  • データ契約 = 社内 IG Performance DB のスキーマを採用:
    • 口座 × 日サマリ: views / reach / impressions / follows_net / profile_visits / link_taps + 各率（like / comment / save / share / pvr / link_ctr / non_follower_reach）+ 視聴層属性（年齢 / 性別 / 国）・流入元・初速 → 口座レベルの方向性・得られている視聴層（#0 が読む）
    • 投稿単位（Reels / Stories / Highlights）: 全メトリクス + コンテンツ本体（hook / body / cta テキスト）+ 型分類（hook_type / cta_type）+ 評価（hook_eval / body_eval / cta_eval / bottleneck / verdict）→ 投稿単位の「何が効いた / 効かない」（#2 / #3 が読む・書く）
    • 共通語彙（Definitions）: Hook 4 型（好奇心扇動 / 逆張り / 問題提起 / FOMO）・CTA 5 型（プロフ誘導＝アフィ本命 / リーチ増幅＝Imp 稼ぎ / 保存シェア / セッション継続 / フォロー）・ファネル TOFU / MOFU / BOFU。Session はこの語彙で分類・推論する
    • 手動ドロップ（CV 側）: ASP の Click 数 / CV 数（発火前に人間が渡す）。
  • 二層: 上流（IG メトリクス）= DB から自動。CV / 売上 = 手動ドロップ。
  • アトリビューションの seam（#4 の課題）: IG 側リンククリック → ASP click → CV（手動）。CV を投稿 / 期間に紐付ける整形が要る。

5. 配管 feasibility（投稿側・確定）

  • 投稿 = GREEN。 社内 Post Tracker に行が乗り、行単位の Access Token を使って設定時刻に API が自動投稿。Session は動画ドラフトリンクをセルに書くだけで投稿成立 → headless 完全対応。最難関がインフラ側で解決済み。
  • 一回限りの人間 setup（発火前）: 新規 IG 垢作成 + API Access Token 発行 + Tracker 行プロビジョン。
  • 動画生成 + ホスティング: Fal.ai API で生成 → リンク化 → 投稿キュー投入。要構築。

6. 残ギャップ / リスク（severity 順）

  1. CV / 売上の読み戻し = 手動 seam で解決（要運用規律）。 ASP に API がないため、各サイクル発火前に人間が手動ドロップ。閉ループは成立するが、ドロップを忘れると CV 信号が止まる → 運用規律と cadence 同期が前提。
  2. headless cron での MCP auth 生存性（未検証）。対話認証が要るツールは無人で落ちる可能性 → 既存 IG 引きで代替検討。
  3. コールド新規垢の 1 ヶ月 reach（売上を主指標にしない理由）。
  4. 動画生成 + ホスティング pipeline 未構築。
  5. コンプラ blast radius（新規垢ゆえ equity リスクは低い）。

7. 運用 setup（発火前の人間タスク）

→ #5「初期セットアップ要件」に統合（アカウント準備 / Git 新規 Repo + 構造 / 自動投稿の接続 / 発火日 + 担当者 / Team 合意）。各項目の担当者割り当てが発火前の必須タスク。

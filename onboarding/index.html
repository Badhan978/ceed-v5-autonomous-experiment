Ceed AI KOL 事業 — 全体像オンボーディング

AI生成のショート動画を、多数の「AI KOL（AIインフルエンサー）」アカウントで量産・投稿し、収益化する運営サイクルの全体地図 ／ 対象: 新メンバー と そのClaude
🎯 最終ゴール：各アカウントで アフィリエイト売上 月5万円 を達成する

すべての施策・改善は「これは月5万円に効くか？」を基準に評価します。単発で終わらせず、円環（分析→戦略→シナリオ→動画→投稿→計測→還流）のどこに効くかを必ず問います。

🟣 一言でいうと

3つのAI（戦略・制作・分析）がバトンを渡し合い、動画を作って投稿し、その結果を学んで次に活かす——この「ぐるぐる回る輪（円環）」を回して稼ぐ事業です。

まずこれだけ押さえればOK

  • 登場するのは3つのAIエージェント。 ① Agent1=戦略（テーマを決める）／ ② Agent2=動画制作（このリポ content-pipeline が担当）／ ③ Agent3=分析（結果を学びに変える）。
  • 輪っかになっている。 戦略→動画→投稿→分析→（学びが戻る）→戦略… と1周ごとに賢くなる。
  • 各エージェントは「スキル」で動く。 一番大きいのは Agent2 の video-creation-orchestrator（シナリオ作成〜投稿予約を自動化）。
  • 人間は要所だけ。 戦略・レビュー・承認は人間（Badhan / 駿冴 / Lala）。それ以外は Mac mini 上で自動で回る。
  • 合言葉は「月5万円に効くか」と「育つもの（自社の資産）に投資」。

1. 背景・目標（なぜ）

やりたいこと

たくさんの AI KOL アカウントで AI製ショート動画を継続投稿し、最終的に アフィリエイト（商品紹介）で売上を立てる。その運営を、人手をかけず 3つのAIエージェントの円環で半自動的に回すのが事業の核です。

なぜ「円環」なのか

動画を作って終わりにせず、投稿結果を分析し、学びを次の戦略・シナリオに戻すことで、回すほど賢くなる構造を狙っています。

🧭 判断の「ものさし」：すたれるもの vs 育つもの

・すたれるもの＝AIが進化すれば誰でもできる作業（動画を作る配管など）→ 薄く・最小限に。
・育つもの＝自社にしか貯まらない資産（投稿結果データ・何が売れたか・Hookや型の辞書・学びの台帳）→ ここに投資する。

2. 3つのエージェントと、その中で動くスキル

① Agent1 — CEO（戦略）

戦略に沿って投稿テーマを絞り込んで確定する。
ceo-run
  • 出力: アカウント一覧シート（戦略・テーマ・投稿時間）
  • 担当: Badhan

② Agent2 — 動画制作

戦略を受けてシナリオ→動画→投稿予約まで実行。このリポが担当。
video-creation-orchestratorscenario-experimentmanual-rule-input
  • 出力: マスタータブ / 投稿管理シート
  • 担当: 駿冴

④ Agent3 — 分析

投稿実績を集め、伸びた/伸びない理由を分析し知見化。

  • 出力: 分析DB / 知見（Insights）
  • 担当: Badhan

4. 各スキルの概要

video-creation-orchestrator Agent2本体・最重要

戦略を受けて作業を作り、「サイクル開始 → シナリオ作成 → 動画ビルド → 投稿予約」を自動で進める司令塔。1台のPCで完結し、動画ビルドだけ別プロセスで Playbook を呼びます。シナリオ・動画には自動の品質チェックが入り、合格分だけ次へ。レビュー（合否・指摘）を受けて学びを記録し、自分の仕様も改善します。

scenario-experiment シナリオ品質の対照実験

「動画の質を決めるのは上流のシナリオ」という考えで、シナリオ生成だけを対象に、1つだけ条件を変えて比べる対照実験を回すスキル。評価軸は 再生数（Imp）最大化。

manual-rule-input 人間のルールを直接登録

人間が言語化したルールを、対話で重複・矛盾を確認しつつ「絶対ルール」へ直接登録。明示承認を経て反映します。

karomi-weekly-report （円環とは独立）

外部クライアント向けの週次レポートを再生成する独立スキル。AI KOLの円環とは別系統です。

5. 知見をどう管理しているか

学びをルールに育てる仕組み（fb-memory）

レビューのフィードバックを「ためる → 似たものをまとめる → 人間（Badhan）が承認 → 絶対ルールに昇華 → 次回のシナリオ/動画に必ず反映」で回します。Agent3の知見もこの流れに合流します。

ためる  →  まとめる  →  人間が承認  →  絶対ルール化  →  次回に反映

知見台帳（L8）

散らばっていた知見を共通IDで1件にまとめ、「古くなったら失効・再検証する」管理を付けた仕組み。古い前提での誤判断を防ぎます。実装・稼働済み

6. 運用環境（どこで動くか）

  • 本番はすべて 1台の Mac mini 上。定期ジョブと常駐プロセスが自動で回り、人手は要所（戦略・レビュー・承認）だけ。
  • 常駐している処理は勝手に止めない・消さない。 動画作成や分析がバックグラウンドで動いており、止めると本番が壊れます。
  • 再起動には注意。 別ユーザーがログインしたままだと定期ジョブが読み込まれないことがあります（担当者がceedでログインし直すと復旧）。

7. いま進めている構造改善

スキル単体の調整から、円環の構造そのものを変える段階へ。改善案を整理し、設計書を別フォルダ「構造改善 設計ドキュメント」に1案ずつ蓄積中。

主な改善案内容状態
CV紐付け「どの売上が・どの投稿から来たか」を結ぶ。月5万円に最も直結最優先・データ準備待ち
知識ライフサイクル知見を名寄せして失効管理（上記の台帳）実装・稼働済み
双方向フィードバック戦略の判断中に「その場の問い」を分析側へ投げ、答えをもらう設計済み・未実装

進め方は「1案ずつ：正当性検証 → 小分けで認識合わせ → 設計を仕上げ → 反対意見で検証 → HTML1枚にしてDriveへ蓄積」。設計書は下のリンク集のフォルダにあります。

8. 気をつけること・働き方の約束

  • 担当領域を越えない。 Agent2（駿冴）/ Playbook（トモさん）/ Agent1・Agent3（Badhan）で担当が分かれています。相手の領域を勝手に変えず、まず自分の担当を確認。
  • 人間専用の欄に書き込まない。 合否やフィードバックなど「人間が判断して記入する欄」へAIが書くのは厳禁です。
  • 分析と判断を混ぜない。 事実・データを出すのは分析側、「だからこうする」と決めるのは人間（CEO）。
  • 勝手に進めず、合意してから。 取り消せない操作（削除など）や相手の領域に関わる変更は先に確認。承認をもらったらその場で実行（放置しない）。
  • 説明は短く小分けに。 情報を一気に詰め込まず、要点を先に。各作業は実測・実読みで都度 verify し、推測で進めない。
  • 完了報告の前に「やり残し」を確認。 自分が回した分だけ見て「全部完了」と言わない。全体を見て未対応が無いか必ずチェック。
  • 日時はすべて日本時間（JST）で扱う。 外部データがUTCなら必ず換算してから集計・表示。
  • このリポはGit運用に注意。 バックグラウンドの自動コミットが走るため、自分の変更はadd〜commitを続けて行うか別ブランチで。対話中のmainへの直接pushは仕組み上ブロックされます（自動pushで後から反映）。
  • 正典（SSOT）を正とする。 このページは要約です。作業時は各リポの正典ファイルを必ず確認してください。
